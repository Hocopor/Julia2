# 04-codex-build-spec.md

## Назначение документа
Этот документ фиксирует **как именно Codex должен технически собрать лендинг**, чтобы на выходе получилась **максимально точная копия референса**, описанного в:
- `01-reference-map.md`
- `02-visual-spec.md`
- `03-content-and-assets.md`

Если предыдущие документы отвечают на вопросы **что строить**, **как это должно выглядеть** и **какой контент использовать**, то этот документ отвечает на вопрос **как именно реализовать страницу в коде**.

Главная цель: не «сделать красивый лендинг в похожем стиле», а **собрать воспроизводимую, аккуратную, предсказуемую копию**.

---

## 1. Приоритет документов
При сборке лендинга Codex обязан использовать документы в следующем порядке приоритета:

1. `01-reference-map.md` — источник истины по структуре, составу блоков и порядку секций
2. `02-visual-spec.md` — источник истины по визуалу
3. `03-content-and-assets.md` — источник истины по контенту и ассетам
4. `04-codex-build-spec.md` — источник истины по технической реализации

### Правило разрешения конфликтов
Если в ходе сборки возникает противоречие:
- **структура и порядок блоков** определяются по `01-reference-map.md`;
- **визуальное решение** определяется по `02-visual-spec.md`;
- **тексты и изображения** определяются по `03-content-and-assets.md`;
- **способ кодовой реализации** определяется по этому документу.

Если какой-то нюанс не описан явно, Codex должен выбирать решение, которое **минимально влияет на визуальное и структурное совпадение с референсом**.

---

## 2. Жёсткое правило задачи
Codex должен понимать задачу буквально:

**Нужно собрать копию лендинга по референсу.**

Не нужно:
- улучшать UX;
- добавлять новые продающие блоки;
- переосмыслять структуру;
- заменять taplink-эстетику на «современный маркетинговый лендинг»;
- использовать дизайнерские вольности;
- оптимизировать структуру «по своему вкусу».

Задача Codex — **воспроизвести**, а не «сделать лучше».

---

## 3. Итоговый ожидаемый результат
На выходе должен получиться:
- одностраничный лендинг;
- один маршрут страницы;
- mobile-first композиция;
- узкая центрированная колонка на коричневом фоне;
- структура, полностью совпадающая с зафиксированной в документации;
- визуал, максимально близкий к референсу;
- корректная работа на мобильных устройствах и десктопе;
- отсутствие лишних элементов.

### Что считается готовым результатом
Готовый результат — это страница, которую можно открыть и увидеть:
- hero;
- «Обо мне»;
- «С какими проблемами я работаю»;
- «Условия работы»;
- «Контакты»;
- все нужные изображения;
- все повторяющиеся CTA;
- правильную типографику, отступы, цвета и ритм;
- отсутствие блоков, которых нет в референсе.

---

## 4. Рекомендуемый стек
Для этой задачи Codex должен использовать **максимально простой и предсказуемый стек**.

### Базовая рекомендация
- **Next.js**
- **TypeScript**
- **App Router**
- **CSS Modules** или один аккуратный глобальный CSS-файл

### Почему именно так
Этот стек:
- позволяет быстро собрать статичный лендинг;
- не навязывает тяжёлую архитектуру;
- хорошо подходит для одной страницы;
- позволяет аккуратно работать с локальными ассетами;
- не заставляет внедрять лишние UI-библиотеки.

### Что допустимо
Если проект уже существует в другом фронтенд-стеке, допустимо адаптировать реализацию под него, **но только при условии**, что:
- итоговый DOM и вёрстка остаются простыми;
- визуал не уходит от референса;
- не добавляется лишняя инфраструктура.

---

## 5. Что запрещено использовать
Codex не должен усложнять проект.

Запрещено добавлять без прямой необходимости:
- Tailwind, если его ещё нет в проекте;
- сторонние UI-kit библиотеки;
- carousels/sliders;
- animation libraries;
- scroll reveal / parallax / GSAP;
- CMS;
- state management библиотеки;
- form libraries;
- дополнительные икон-паки ради одной бургер-иконки;
- большие theme-системы.

### Принцип
Это **статический визуальный лендинг**, а не приложение.

---

## 6. Техническая модель страницы
Страница должна быть реализована как **один маршрут**:
- `/`

Внутри этого маршрута должен быть один основной компонент страницы с последовательным выводом секций.

### Общая схема DOM
```text
<body>
  <main>
    <div class="page-bg">
      <div class="landing-shell">
        <section class="hero" />
        <section class="about" />
        <section class="issues" />
        <section class="pricing" />
        <section class="contacts" />
      </div>
    </div>
  </main>
</body>
```

### Ключевой принцип
В реальном DOM это должна быть **одна вертикальная колонка**, а не две колонки, не poster-layout и не эмуляция двух телефонов рядом.

---

## 7. Рекомендуемая структура файлов
Ниже — предпочтительная структура проекта, если Codex собирает страницу с нуля внутри Next.js-приложения.

```text
/app
  layout.tsx
  page.tsx
  globals.css

/components
  landing/
    landing-page.tsx
    hero-section.tsx
    about-section.tsx
    issues-section.tsx
    pricing-section.tsx
    contacts-section.tsx
    section-title.tsx
    cta-button.tsx
    issue-card.tsx
    pricing-card.tsx
    burger-icon.tsx

/lib
  landing-content.ts

/public
  /assets
    hero-photo.webp
    about-photo.webp
    issues-flower.webp
    pricing-flower.webp
    sea-photo.webp
    contacts-photo.webp
```

### Что важно
- структура может быть немного проще;
- но все секции должны быть физически отделены и читабельны;
- весь контент должен быть собран в одном месте, а не размазан хаотично по JSX.

---

## 8. Правило контента
Codex должен использовать **статические данные**.

Для этого рекомендуется вынести тексты, подписи ссылок, названия тарифов и пути к ассетам в отдельный конфиг, например:
- `/lib/landing-content.ts`

### Почему так лучше
Это даёт:
- прозрачность;
- удобство проверки;
- отсутствие случайных правок прямо в разметке;
- лёгкую замену временных ссылок и изображений.

### Пример структуры данных
```ts
export const landingContent = {
  hero: {
    role: 'ПСИХОЛОГ, КПТ-ТЕРАПЕВТ',
    firstName: 'ВЛАДИСЛАВА',
    lastName: 'БИРИХ',
    quote: 'ПОМОГУ РАЗРЕШИТЬ ВНУТРЕННИЕ КОНФЛИКТЫ, РАЗОБРАТЬСЯ В СЕБЕ И СВОИХ ЧУВСТВАХ И ПРИБЛИЗИТЬСЯ К НАСТОЯЩЕМУ СПОКОЙСТВИЮ.',
    ctaLabel: 'ЗАПИСАТЬСЯ НА КОНСУЛЬТАЦИЮ',
    imageSrc: '/assets/hero-photo.webp',
  },
};
```

Это пример подхода, а не жёсткое требование к синтаксису.

## 9. Компонентная карта
Codex должен собирать страницу из простых компонентов.

### 9.1. `LandingPage`
Родительский компонент всей страницы.

Отвечает за:
- общий фон;
- центрирование колонки;
- последовательный рендер всех секций.

### 9.2. `HeroSection`
Содержит:
- подпись профессии;
- имя;
- бургер-иконку;
- hero-фото;
- белую карточку с текстом;
- главную CTA-кнопку.

### 9.3. `AboutSection`
Содержит:
- заголовок;
- абзац;
- фото.

### 9.4. `IssuesSection`
Содержит:
- заголовок;
- карточки проблем;
- декоративную вставку-цветок между карточками 2 и 3.

### 9.5. `IssueCard`
Переиспользуемый компонент карточки проблем.

Должен поддерживать как минимум:
- title;
- description;
- number;
- theme: `light | dark`.

### 9.6. `PricingSection`
Содержит:
- заголовок;
- верхнее фото цветка;
- три тарифных блока;
- изображение моря между первым и вторым тарифом;
- маленький поясняющий текст.

### 9.7. `PricingCard`
Переиспользуемый компонент тарифа.

Должен принимать:
- number;
- title;
- durationLabel;
- durationValue;
- priceLabel;
- priceValue;
- ctaLabel;
- href.

### 9.8. `ContactsSection`
Содержит:
- фото;
- заголовок поверх фото;
- нижнюю полосу ссылок.

### 9.9. `CTAButton`
Переиспользуемый компонент всех кнопок записи.

Все CTA на странице должны использовать **один визуальный стиль**.

### 9.10. `BurgerIcon`
Небольшой декоративный компонент.

Может быть реализован:
- через inline SVG;
- через span-элементы;
- через псевдоэлементы CSS.

Иконка **не обязана открывать меню** на первом этапе.

---

## 10. Семантическая разметка
Семантика должна быть аккуратной, но без фанатизма.

Рекомендуется:
- один `h1` на странице — имя специалиста в hero;
- остальные крупные заголовки секций — `h2`;
- названия карточек и тарифов — `h3` или стилизованный текст в зависимости от структуры;
- основное содержимое оборачивать в `main`;
- блоки страницы делать через `section`.

### Важно
Семантика не должна ломать визуал. Если ради точного совпадения нужен более нейтральный тег, допустимо использовать его, но с сохранением доступности.

---

## 11. Правила по ссылкам и интерактивности
На странице реально интерактивными должны быть только:
- CTA-кнопки;
- ссылки в блоке контактов.

### CTA-кнопки
До получения реальных ссылок Codex может использовать:
- `href="#contact"`
или
- временный заглушечный URL из конфигурации.

### Контакты
До получения реальных адресов допустимы временные ссылки:
- `#`
- `https://wa.me/placeholder`
- `https://t.me/placeholder`
- `#development`

### Что важно
Даже при заглушечных ссылках:
- подписи должны быть финальными;
- layout не должен меняться после подстановки настоящих ссылок.

---

## 12. Реализация макета

## 12.1. Общий холст
Codex должен реализовать внешний фон страницы отдельно от самой узкой колонки.

Нужно:
- задать коричневый фон всей странице;
- в центре разместить бело-молочную колонку лендинга;
- не растягивать контент на всю ширину десктопа.

### Рекомендуемая логика
- `body` / `html`: фоновый коричневый цвет;
- отдельный контейнер: центрирование по горизонтали;
- верхний и нижний воздух сохранить.

## 12.2. Ширина колонки
Codex должен придерживаться мобильной ширины.

Рекомендованное правило:
```css
width: min(396px, calc(100vw - 24px));
```

Допустимое визуальное отклонение: не более `±8px`.

## 12.3. Радиусы
Радиусы должны быть минимальными.

Допустимо:
- `0px`
- `1px`
- `2px`

Большие скругления запрещены.

## 12.4. Тени
Сильных теней быть не должно.

Допустимо:
- отсутствие теней;
- едва заметная техническая тень, если иначе блок теряется на фоне.

Но визуально лендинг должен оставаться почти плоским.

## 13. Реализация секций

## 13.1. Hero
Hero должен быть собран так, чтобы:
- имя располагалось выше фотографии;
- бургер находился в правом верхнем углу блока;
- карточка с текстом перекрывала нижнюю часть hero-фото;
- CTA была сразу под карточкой.

### Техническая рекомендация
Для hero удобно использовать:
- контейнер секции;
- отдельную обёртку фотографии с `position: relative`;
- текстовую карточку через `position: relative` и отрицательный `margin-top`, либо через абсолютное позиционирование внутри photo-wrapper.

### Что важно
Карточка не должна уезжать слишком низко и не должна выглядеть как отдельная секция. Она должна восприниматься как часть hero-композиции.

---

## 13.2. About
Секция должна быть простой.

Реализовать:
- центрированный заголовок;
- узкий текстовый контейнер;
- фото снизу.

Важный принцип:
- текст не должен растягиваться на всю ширину секции;
- фото должно быть шире текстового блока.

---

## 13.3. Issues
Эта секция требует самого аккуратного соблюдения визуального ритма.

Codex должен:
- создать тёмный внутренний блок на коричневом фоне страницы;
- обернуть всю секцию тонкой светлой рамкой;
- выстроить карточки в одну колонку;
- реализовать чередование светлой и тёмной карточек;
- вставить декоративное фото цветка между карточками `/2` и `/3`.

### Важно
Карточки должны выглядеть как модули одной системы, а не как разные дизайнерские блоки.

---

## 13.4. Pricing
Pricing-секция должна быть модульной и повторяемой.

Codex должен:
- сверху поставить заголовок;
- сразу под ним — фото цветка;
- далее реализовать три повторяющихся тарифных блока;
- между первым и вторым тарифом вставить изображение моря;
- под тарифами вывести мелкий поясняющий текст.

### Рекомендуемая логика компонента тарифа
Каждый тариф состоит из:
1. строки с номером и названием;
2. двух параметрических строк;
3. CTA.

### Параметрические строки
Это не обычный текст абзацем. Это именно отдельные прямоугольные строки с рамкой.

---

## 13.5. Contacts
Секция должна быть собрана как:
- изображение;
- заголовок поверх изображения;
- нижняя светлая полоса ссылок.

### Техническая рекомендация
Сделать общий блок с изображением и overlay-заголовком.

Нижнюю панель ссылок реализовать отдельным контейнером под изображением.

Важно:
- ссылки должны помещаться в одну строку;
- они не должны превращаться в массивные кнопки;
- по стилю это почти footer-tabline, а не CTA-группа.

---

## 14. Работа с изображениями

## 14.1. Источник изображений
Codex должен использовать только те ассеты, которые перечислены в `03-content-and-assets.md`.

### Пути
Ожидаемые пути:
- `/assets/hero-photo.webp`
- `/assets/about-photo.webp`
- `/assets/issues-flower.webp`
- `/assets/pricing-flower.webp`
- `/assets/sea-photo.webp`
- `/assets/contacts-photo.webp`

## 14.2. Способ вставки
Допустимо использовать:
- `next/image`
- обычный `img`

### Практическая рекомендация
Если `next/image` мешает точному визуальному совпадению по кропу, допускается использовать обычный `img`.

Приоритет — **точный визуал**, а не формальная «правильность» инструмента.

## 14.3. Кадрирование
Для всех изображений важны:
- `object-fit: cover` там, где нужно сохранять модульную форму;
- фиксированные или полуфиксированные пропорции модулей;
- отсутствие случайного растягивания.

## 14.4. Обязательная стабильность
После замены временных изображений на финальные:
- не должен ломаться layout;
- не должны скакать высоты секций;
- не должны уезжать карточки и кнопки.

## 15. Правила CSS

## 15.1. Подход к стилям
Нужно писать спокойный, прямой CSS.

Подходит:
- CSS Modules;
- один глобальный stylesheet + секционные классы.

Не подходит:
- хаотичный инлайн-стайлинг;
- длинные цепочки utility-классов ради utility-классов;
- смешение нескольких несовместимых стилистических систем.

## 15.2. Именование
Классы должны быть понятными.

Примеры:
- `pageBg`
- `landingShell`
- `heroSection`
- `heroImage`
- `quoteCard`
- `issuesCardDark`
- `pricingMetaRow`
- `contactsLinks`

## 15.3. Типографика
Шрифтовая реализация должна следовать `02-visual-spec.md`.

Если точный шрифт референса недоступен, использовать сочетание:
- для display-части: элегантный serif;
- для основного текста: нейтральный sans-serif или мягкий serif, в зависимости от лучшего совпадения.

### Приоритет
Приоритетом является **визуальное сходство**, а не буквальная привязка к конкретной гарнитуре.

## 15.4. Цвета
Codex должен взять базовые цвета из `02-visual-spec.md` и при необходимости сделать **минимальную тонкую подстройку**, а не произвольную замену палитры.

---

## 16. Адаптив
Лендинг строится по принципу **mobile-first**.

### 16.1. Мобильная версия
Это главная версия страницы.

На мобильных устройствах:
- колонка почти заполняет экран по ширине;
- сохраняются небольшие боковые поля;
- вертикальный ритм не ломается;
- все элементы читаются без сжатия.

### 16.2. Планшет и десктоп
На широкой ширине экрана:
- лендинг остаётся узким;
- не превращается в широкий сайт;
- не получает второй колонки;
- не разъезжается по центру.

### 16.3. Что нельзя делать на десктопе
Запрещено:
- раскладывать секции в две колонки;
- ставить рядом «Обо мне» и «Условия работы»;
- превращать страницу в magazine-layout;
- повторять постерную композицию с двумя телефонами.

---

## 17. Минимальные технические требования к качеству
Страница должна:
- корректно открываться без ошибок сборки;
- не иметь битых путей к ассетам;
- не иметь переполнений по ширине;
- не иметь неожиданного горизонтального скролла;
- не иметь налезающего текста;
- не ломаться при подстановке реальных ссылок;
- не иметь внезапных скачков layout из-за изображений.

---

## 18. Доступность и базовая корректность
Хотя задача прежде всего визуальная, Codex должен соблюдать минимальную корректность.

### Нужно сделать
- у всех изображений должны быть `alt`;
- все интерактивные элементы должны быть реальными ссылками или кнопками;
- hover/focus состояния должны быть мягкими и неброскими;
- контраст текста должен оставаться читаемым.

### Но важно
A11y не должна приводить к визуальному отклонению от референса.

- `lang="ru"`;

**description:**

## 21. Чеклист сборки для Codex
Перед тем как считать задачу выполненной, Codex должен проверить следующее.

### 21.1. Структура
- есть ровно 5 основных секций;
- они идут в правильном порядке;
- внутри каждой секции присутствуют все обязательные элементы.

### 21.2. Контент
- все тексты совпадают с документацией;
- названия тарифов не изменены;
- `РУБЛЕЙ` не заменено на `₽`;
- CTA везде одинаковая;
- ссылки в контактах подписаны корректно.

### 21.3. Визуал
- колонка узкая и центрированная;
- фон коричневый;
- белые и коричневые модули совпадают по настроению;
- карточка-цитата действительно лежит поверх hero-фото;
- issues-секция выглядит как цельный блок с рамкой;
- pricing-секция выглядит модульно и аккуратно;
- contacts-секция закрывает страницу правильно.

### 21.4. Техническая чистота
- нет console errors;
- нет битых импортов;
- нет лишних зависимостей;
- нет мёртвых компонентов, не используемых в странице;
- нет временных отладочных подписей на странице.

## 22. Definition of Done
Задача считается выполненной, если одновременно соблюдены все условия:

1. Страница визуально узнаётся как копия присланного референса.
2. Структура полностью соответствует `01-reference-map.md`.
3. Визуальный язык соответствует `02-visual-spec.md`.
4. Контент и ассеты соответствуют `03-content-and-assets.md`.
5. Реализация не содержит самодеятельности.
6. Лендинг корректно работает как один вертикальный mobile-first экран.
7. На странице нет ничего, чего нет в референсе и документации.

## 23. Финальная инструкция для Codex
Codex должен работать по следующему принципу:

1. Сначала прочитать все 4 документа.
2. Не начинать код, пока не понятна карта секций.
3. Сначала собрать правильный каркас страницы.
4. Затем выставить размеры, отступы и цвета.
5. Затем подключить тексты и изображения.
6. Затем довести визуал до максимально близкого совпадения.
7. В конце удалить любую самодеятельность и всё лишнее.

### Главный принцип
Если Codex сомневается между:
- «сделать креативнее»
и
- «сделать ближе к референсу»,

нужно всегда выбирать:
**сделать ближе к референсу**.

## 25. Что должно остаться после завершения работы
После выполнения задачи в проекте должны остаться:
- чистая структура файлов;
- один собранный лендинг;
- ассеты в правильных путях;
- понятные компоненты;
- стили без хаоса;
- страница без лишних элементов.

Итоговая сборка должна быть такой, чтобы по ней можно было сразу судить: **это именно копия нужного лендинга, а не новая интерпретация**.
